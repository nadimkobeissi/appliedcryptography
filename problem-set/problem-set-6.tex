\section{Design-Level Security and Formal Verification}
\begin{questions}
	\subsection*{Symbolic vs. Computational Models}

	\question[12] \textbf{Protocol Modeling Trade-offs:}\topicbadge{2.5}
	You are tasked with formally verifying a new secure messaging protocol for a government agency. The protocol involves complex key derivation, multiple handshake modes, and a large number of Diffie-Hellman operations.
	\begin{parts}
		\part Compare the advantages and limitations of using symbolic (ProVerif) versus computational (CryptoVerif) approaches for this protocol. Which properties would be easier to verify in each model?
		\part The protocol uses a custom authenticated encryption scheme. Explain, at a high level, how such a construction would be modeled in the symbolic model. What assumptions would you need to make?
	\end{parts}

	\question[12] \textbf{Equivalence Properties in Practice:}\topicbadge{2.5}
	A voting protocol claims to provide receipt-freeness: voters cannot prove how they voted, even if they want to.

	\textbf{Background:} Recall that security properties in formal verification can be classified as:
	\begin{itemize}
		\item \textbf{Trace properties}: Properties that can be verified by examining individual execution traces (e.g., ``the secret key is never sent in plaintext'')
		\item \textbf{Equivalence properties}: Properties that require comparing multiple executions (e.g., ``an attacker cannot distinguish between two different scenarios'')
	\end{itemize}

	Receipt-freeness means that even if Alice wants to prove she voted for candidate A (perhaps under coercion), she cannot produce evidence that distinguishes her voting for A from voting for B. This inherently involves comparing two scenarios.

	\begin{parts}
		\part[6] Explain why receipt-freeness is naturally formulated as an equivalence property rather than a trace property.\footnote{Hint: Consider what distinguishes trace properties (examining individual executions) from equivalence properties (comparing multiple scenarios).} Then formulate receipt-freeness as an equivalence property in the symbolic model.\footnote{Hint: Think about what two processes should be indistinguishable to model this property.}
		\part[6] The slides contrast two ways of formulating secrecy in the symbolic model:\footnote{\url{https://appliedcryptography.page/slides/\#2-5}}
		\begin{itemize}
			\item \textbf{As a trace property}: The secret value never appears in the adversary's knowledge.
			\item \textbf{As an equivalence property}: The adversary cannot distinguish the real secret from a random value.
		\end{itemize}
		\begin{enumerate}
			\item Explain why the equivalence formulation is strictly stronger than the trace formulation.
			\item Describe a protocol that satisfies the trace property but violates the equivalence property.\footnote{Hint: The protocol does not need to reveal the secret itself - only something derived from it.}
			\item Give a concrete scenario in which this gap enables a practical attack.
		\end{enumerate}
	\end{parts}

	\end{parts}
\end{questions}

\section{Implementation Security and Side Channels}
\begin{questions}
	\subsection*{Constant-Time Implementation}

	\question[8] \textbf{Compiler and Hardware Challenges:}\topicbadge{2.5}
	AES implementations commonly use lookup tables (S-boxes) for the SubBytes operation, where table indices depend on secret key material XORed with plaintext. This creates timing variations through cache hits/misses that can leak key bits.

	You've written constant-time code for AES, carefully avoiding these vulnerabilities by eliminating table lookups, key-dependent branches, and variable-time operations.
	\begin{parts}
		\part[4] Give three specific examples of how a modern compiler might destroy your constant-time guarantees through ``optimization.'' For each, show the original code and the problematic transformation.
		\part[4] Even with perfect constant-time assembly code, modern CPUs introduce timing variations. The slides caution that constant-time proofs hold only \textit{relative to a leakage model}: hardware may leak more than the model captures, and physical attacks bypass software protections entirely.
		\begin{enumerate}
			\item Explain how branch prediction and speculative execution (as in Spectre-style attacks) can leak secrets even when code satisfies the constant-time model.
			\item The TEE.fail attack (2025) physically interposes on the DDR5 memory bus to extract secrets from trusted execution environments. Explain why physical attacks of this kind fall outside all three leakage models covered in the course, and what this implies about the scope of formal side-channel guarantees.
		\end{enumerate}
	\end{parts}

	\subsection*{Side-Channel Resistance}

	\question[8] \textbf{Understanding Leakage Models:}\topicbadge{2.5}
	You're implementing cryptographic primitives and need to ensure they're resistant to side-channel attacks.
	\begin{parts}
		\part[4] Explain the three main leakage models covered in the course: program counter (PC) model, constant-time model, and size-respecting model. For each model, describe what information leaks and give a concrete example of a coding pattern that would violate it.
		\part[4] Consider this password checking code:
		\begin{verbatim}
for (i = 0; i < len; i++) {
    if (password[i] != input[i]) {
        return FAIL;
    }
}
return SUCCESS;
\end{verbatim}
		Explain why this violates constant-time principles and show how to rewrite it to be constant-time. What is the performance trade-off?
	\end{parts}
\end{questions}

\section{Computer-Aided Cryptography in Practice}
\begin{questions}
	\subsection*{The Verification Boundary}

	\question[30] \textbf{Auditing a ``Formally Verified'' Library:}\topicbadge{2.5}
	A startup advertises its cryptographic library as ``formally verified,'' without qualification. An independent audit - much like the libcrux audit presented in the slides - uncovers the following findings:
	\begin{enumerate}
		\item The library's public decryption API unconditionally unwraps the result returned by the verified decryption core, so any tampered ciphertext crashes the calling application.
		\item The verified field arithmetic depends on a platform multiplication intrinsic whose specification is axiomatized: the verifier accepts it without proof. On one supported platform, the intrinsic's software fallback computes incorrect results.
		\item One module's formal specification transcribes a rounding constant incorrectly from the standard it claims to formalize. The conformance proof still ``passes,'' because the build system admits the module by default, so its proof obligations are never actually checked.
		\item The key exchange wrapper feeds the raw Diffie-Hellman output directly into key derivation, omitting the all-zero output check that the relevant RFC mandates.
		\item The signature implementation is provably correct with respect to its formal specification, but the specification omits a normalization requirement that major deployed systems enforce, so some of the library's signatures are rejected in practice.
	\end{enumerate}

	\begin{parts}
		\part[10] Classify each finding according to the slides' taxonomy of verification boundary failures: Type I (unverified platform abstractions), Type II (unverified protocol logic), Type III (specification incompleteness), Type IV (unverified wrapper logic), or Type V (specification errors and false proofs). Briefly justify each classification.
		\part[10] The slides argue that admitted proofs are \textit{more dangerous} than code that was never claimed to be verified. Explain this argument with reference to finding 3. As part of your answer, explain what lax checking does and does not establish (syntax, types, proofs), and describe the mechanisms that allow a false proof to survive in a codebase advertised as formally verified.
		\part[10] The startup hires you to repair its verification process. Drawing on the slides' five recommendations and the boundary-management practices of projects such as seL4, CompCert, and AWS's verified libcrypto, propose a concrete remediation plan. For each of the five findings, identify which element of your plan would have caught or prevented it.
	\end{parts}

	\subsection*{Building Verified Systems}

	\question[20] \textbf{End-to-End Verification Project:}\topicbadge{2.5}
	You're leading the verification of a new secure communication protocol that will be deployed at nation-state scale. The protocol combines classical and post-quantum cryptography, supports multiple authentication modes, and must resist both computational and side-channel attacks.
	\begin{parts}
		\part[5] \textbf{Verification Architecture:}
		Design a comprehensive verification strategy using multiple tools. Specify which properties each tool will verify and identify gaps between different verification layers.

		\part[5] \textbf{Formal Specifications:}
		Write formal security definitions for the protocol's goals. Specify side-channel resistance requirements formally.

		\part[5] \textbf{Implementation Strategy:}
		Choose implementation languages and verification tools. Design the development workflow to maintain verification as the protocol evolves, ensuring that all proofs are actually re-checked on every change rather than silently admitted.

		\part[5] \textbf{Assurance Arguments:}
		Create a comprehensive assurance case for stakeholders. Document the verification boundary explicitly, addressing the ``fine print'' - what's not verified and why. Using the slides' spectrum of verification claims, state which claim your system could honestly make and how you would communicate its scope without engaging in verification theatre.
	\end{parts}
\end{questions}
